  <action>
Create NLPService class with parse(text: str, user_timezone: str, context_date: datetime) -> ParseResult method.

ParseResult dataclass should contain:
  - title: str (extracted event name)
  - start_at: datetime (parsed start, in user's timezone initially, will be converted to UTC by caller)
  - end_at: datetime | None (parsed end, or None if missing)
  - confidence_date: float (0.0-1.0, how confident in date parsing)
  - confidence_title: float (0.0-1.0, how confident in title extraction)
  - recurrence: dict | None (e.g., {"freq": "WEEKLY", "count": 10} or None for one-time)
  - errors: list[str] (validation errors encountered during parse, if any)
  - raw_text: str (original input, preserved for context)

NLPService implementation must:
- Use dateparser library with strict settings: RETURN_AS_TIMEZONE_AWARE=False initially, then apply user_timezone
- Support these phrase classes (from CONTEXT Parsing acceptance rules):
  * Explicit: "2026-04-10 14:00", "15/03/2026 2pm"
  * Relative: "tomorrow", "tomorrow at 4pm", "in 3 days", "next Friday 2pm"
  * Week-relative: "next Monday", "last Friday"
  * Recurrence: "every Friday 2pm", "every 2 weeks", "daily"
  * Month/day: "March 15 2pm", "December 25"

- For ambiguous numeric dates (e.g., 03/04/2026 with DD/MM mode):
  * Always interpret as DD/MM without confirmation prompt
  * Set confidence_date = 0.95 (not 1.0, indicates fixed interpretation)

- For month/day without year (e.g., "March 15"):
  * Auto-pick nearest future date relative to context_date
  * If result is past, pick next year's occurrence
  * Set confidence_date = 0.85

- For date without time (e.g., "dentist Friday"):
  * Parse date correctly
  * Default time to 09:00 (from CONTEXT Review form behavior)
  * Set confidence_date = 1.0 (date certain) but add annotation

- For conflicting date/time (e.g., "Friday March 10" when March 10 is Tuesday):
  * Detect mismatch and add error: "Conflicting day/date: March 10 is not Friday in 2026"
  * Return ParseResult with errors list populated, confidence_date = 0.0

- For past-only dates (e.g., today is March 19, user says "March 10 2pm"):
  * Detect past and add error: "Date is in past: 2026-03-10 < 2026-03-19"
  * Return ParseResult with errors list, confidence_date = 0.0

- For phrases with no date anchor (e.g., "meeting at 2pm" with no date):
  * Add error: "No date found in input"
  * Return ParseResult with errors list, confidence_date = 0.0

- For recurrence phrases:
  * Parse keywords: "every", "daily", "weekly", "monthly", "yearly", "times", "occasions"
  * Support patterns: "every Friday 2pm", "every 2 weeks", "daily for 10 times"
  * Map to FREQ (DAILY, WEEKLY, MONTHLY, YEARLY), COUNT, INTERVAL
  * Scope limit: common patterns only (daily/weekly/monthly/yearly and every N weeks per CONTEXT)
  * Set recurrence dict if detected, else None

- Timezone handling:
  * Input: user_timezone (e.g., "America/New_York")
  * Parse date/time using user's timezone as anchor
  * Do NOT show "UTC" or "timezone" in any error messages (per CONTEXT Ambiguity handling)
  * Return start_at and end_at ready for UTC conversion by caller

- Error handling:
  * Catch dateparser exceptions gracefully, return ParseResult with errors list
  * Add field-level confidence flags so frontend knows what to highlight

Call validate_rrule() (imported from app.events.recurrence) on any parsed RRULE before returning to catch RFC5545 issues early.

Test against examples from CONTEXT Parsing acceptance rules section.
