1. Create `app/licensing/__init__.py` (empty).

2. Create `app/licensing/keys.py` with two functions:
   - `generate_license_key(secret: str) -> str`:
     - Generate 18 random bytes using `secrets.token_bytes(18)`
     - Compute HMAC-SHA256 of those bytes with `secret` as key
     - Take first 6 bytes of the HMAC digest
     - Encode the 18 payload bytes + 6 signature bytes as base32 (no padding)
     - Format as `SYNCO-XXXXXX-XXXXXX-XXXXXX-XXXXXX` (split into 4 groups of 6 chars from 24 payload chars, then 2 groups of 5 from 10 sig chars — simplified: just format the base32 into dash-separated groups of 5-6 chars)
     - Simpler approach: use `secrets.token_hex(16)` for 32 hex chars as payload, compute `hmac.new(secret.encode(), bytes.fromhex(payload), 'sha256').hexdigest()[:8]` as check, format as `SYNCO-{payload[:8]}-{payload[8:16]}-{payload[16:24]}-{payload[24:32]}-{check}`
   - `validate_license_key(key: str, secret: str) -> bool`:
     - Parse the key, extract payload and check portions
     - Recompute HMAC and compare using `hmac.compare_digest`
     - Return True if valid, False otherwise
     - Handle malformed keys gracefully (return False)

3. Create `app/licensing/generate.py` as a CLI entry point:
   ```python
   """Generate a Synco self-hosted license key.

   Usage: python -m app.licensing.generate <secret>
   """
   import sys
   from app.licensing.keys import generate_license_key

   def main():
       if len(sys.argv) < 2:
           print("Usage: python -m app.licensing.generate <secret>")
           sys.exit(1)
       secret = sys.argv[1]
       key = generate_license_key(secret)
       print(f"License Key: {key}")

   if __name__ == "__main__":
       main()
   ```
   Also create `app/licensing/__main__.py` that calls `generate.main()` so `python -m app.licensing <secret>` works.

4. Create `app/licensing/middleware.py` with `LicenseCheckMiddleware(BaseHTTPMiddleware)`:
   - On startup (first request or `__init__`), read `Settings().SYNCO_LICENSE_KEY` and `Settings().SYNCO_LICENSE_SECRET`
   - If `ENVIRONMENT` is NOT `self-hosted`, skip all license checking (pass through)
   - If `ENVIRONMENT == "self-hosted"`:
     - If `SYNCO_LICENSE_KEY` is empty: set `self.license_valid = False`
     - If key exists: validate with `validate_license_key(key, secret)`, set `self.license_valid` accordingly
   - In `dispatch()`: if `self.license_valid is False` AND the response is HTML (Content-Type contains text/html), inject a warning banner before `</body>`:
     ```html
     <div style="position:fixed;bottom:0;left:0;right:0;background:#dc2626;color:white;text-align:center;padding:12px;font-size:14px;z-index:9999;">
       ⚠️ Invalid or missing license key. Please set SYNCO_LICENSE_KEY in your .env file. <a href="https://synco.app/pricing" style="color:#fef08a;text-decoration:underline;">Purchase a license</a>
     </div>
     ```
   - API/JSON responses are NOT modified (only HTML)
   - `/health` and `/health/ready` are always passed through without modification

5. Add to `config.py` Settings class:
   - `SYNCO_LICENSE_KEY: str = ""`
   - `SYNCO_LICENSE_SECRET: str = ""`

6. Add `LicenseCheckMiddleware` to `main.py` — add it as the outermost middleware (before SecurityHeadersMiddleware) so it can modify HTML responses:
   ```python
   from app.licensing.middleware import LicenseCheckMiddleware
   app.add_middleware(LicenseCheckMiddleware)
   ```

7. Update `.env.example` with new variables:
   ```
   # === Self-Hosted License ===
   SYNCO_LICENSE_KEY=           # License key from purchase (self-hosted only)
   # SYNCO_LICENSE_SECRET=      # Admin-only: secret for generating keys
   ```

   NOTE: The Supabase self-hosted setup is complex (requires kong, gotrue, postgrest, etc.). For a simpler self-hosted approach, use direct PostgreSQL with Supabase client pointed at the local PostgREST. However since the app uses `SupabaseStore` which relies on the Supabase client, we need a compatible API. The simplest approach: use `supabase/postgres` + `postgrest/postgrest` as the database layer, and configure `SUPABASE_URL` to point at the local PostgREST API.

   Actually, re-evaluate: the app uses `SupabaseStore` which wraps the Supabase Python client. For self-hosted, the simplest path is to use the official Supabase self-hosted stack (supabase/supabase docker). BUT that's very heavy (10+ containers). Instead, use a minimal stack:
   - `supabase/postgres:15` for the database
   - `postgrest/postgrest` for the REST API (Supabase client talks to this)
   - Point `SUPABASE_URL` at `http://postgrest:3000` and `SUPABASE_ANON_KEY` at a locally-generated JWT

   Even simpler: since the app's `SupabaseStore` uses the `supabase-py` client which talks to PostgREST, we can use the official supabase self-hosted minimal setup. Use `supabase/studio` image is optional. The minimum viable services are: postgres + postgrest + gotrue (for auth).

   **Final approach for v4.0:** Use the Supabase CLI local dev setup pattern but for production:
   - postgres (supabase/postgres)
   - postgrest (postgrest/postgrest)
   - Configure with a locally-generated JWT secret for `SUPABASE_ANON_KEY`

   Write the docker-compose.yml with these 4 services: app, db (postgres), rest (postgrest), caddy.
